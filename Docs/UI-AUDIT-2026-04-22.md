## 4. Decisão estratégica: Next.js vs. manter Vite

A skill declara **"Stack obrigatória: Next.js (App Router) + Tailwind + shadcn/ui + Framer Motion + lucide-react"**. O projeto está em **Vite**. Isso cria uma bifurcação que você precisa decidir antes de começar.

### Opção A — Migrar para Next.js (100% aderente à skill)

**Prós:**
- Aderência total à skill, incluindo App Router com layouts por área (`/engenharia/layout.tsx` força `dark`, `/cartorio/layout.tsx` força light)
- `next/font/google` para Inter com preload otimizado
- Server Components permitem esconder lógica sensível (se um dia houver MySQL no frontend-adjacent)
- Caminho natural pra SEO/SSR futuro (landing, prospect)

**Contras:**
- Refatoração de deploy no Railway (hoje usa `serve` estático; Next precisa de Node server, `next start`)
- Re-escrever o único arquivo de app (`App.tsx`) em layouts + páginas
- Build mais lento que Vite
- `railway.json` do `maprix-web` precisa ajuste: build `next build`, start `next start -p $PORT`
- Tempo estimado: **6–10h só de migração**, sem contar design

### Opção B — Manter Vite + adotar 90% da skill (pragmática)

**Prós:**
- Sem risco de quebrar deploy atual (Railway `serve dist`)
- Tailwind, shadcn/ui, Framer Motion, lucide, sonner, next-themes (sim, funciona fora do Next), React Router funcionam igual em Vite
- Migração incremental: fundação primeiro (tokens, fontes, tema), componentes depois, rotas por último
- Tempo estimado: **3–5h** de fundação + trabalho de telas sob medida

**Contras:**
- Divergência nominal com a skill (ela diz "Next.js obrigatório")
- `next/font` substituído por `@fontsource/inter` (ligeiramente menos otimizado, imperceptível)
- Layouts por área via `<Route>` + `<ThemeProvider>` wrapper em vez de `app/<area>/layout.tsx`
- Se futuramente precisar SSR (ex: landing pública indexável), migração ainda vai acontecer — só adiada

### 💡 Recomendação

**Opção B** para este MVP. Justificativa objetiva:

1. O cliente-alvo é **engenheiro/cartório logado**, não SEO público — SSR não agrega valor hoje
2. O Railway já está estável; mudar build durante o ciclo do MVP introduz risco sem ganho imediato
3. 90% do valor da skill (tokens, componentes, animações, estados, checklist §8) é framework-agnóstico
4. Migrar pra Next fica como dívida explícita no backlog, acionável quando landing pública entrar no escopo

Se **você preferir A** (aderência total), sinaliza e eu refatoro. Custo é tempo, não qualidade — ambos os caminhos entregam um produto polido.

## 8. Riscos e observações

- **Lockfile e versão do Tailwind**: Tailwind 4 (recém-lançado) muda a config. A skill assume Tailwind 3. Vou instalar Tailwind 3.4 pra não diverger.
- **shadcn em Vite**: requer config extra em `components.json` com paths `@/`. Resolvível, documentado; faço na Milestone 1.
- **Rebuild do Railway**: cada push da Milestone 1 já dispara redeploy — sem quebra, mas vai consumir ~1min de deploy. Tudo bem?
- **Dual theme forcing**: `next-themes` normalmente usa `storage` do browser; forçar light em `/cartorio` independente da preferência do user pode confundir quem espera dark. Skill é clara: "default respeita o contexto, toggle disponível". Vou implementar um override de rota que **não** persiste (respeita toggle dentro da área). OK?
