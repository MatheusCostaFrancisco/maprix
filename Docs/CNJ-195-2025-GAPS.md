## ✅ GAP-01 — Anel exterior em sentido **counter-clockwise** — RESOLVIDO (commit pendente)

**Item do checklist:** 2.7
**Evidência:** shoelace sum do polígono convertido deu `−1.76e-6` (negativo ⇒ CCW). Spec ESRI Shapefile (rev. jul/1998, seção 3.11) é explícita: *"Vertices of rings defining holes in polygons are in a counterclockwise direction. Vertices of outer rings are in clockwise order."*

**Impacto:** QGIS e muitos leitores aceitam silenciosamente, mas validadores rigorosos (ArcGIS stricts, alguns PostGIS configs, ogr2ogr com `--check-geometry`) tratam como inválido. Sistemas de validação do SIG-RI podem flagar como "geometria inválida" ou — pior — interpretar como **ilha** (furo), resultando em área negativa/zero.

**Causa raiz:** o gerador escreve pontos exatamente na ordem recebida. Após `convertPolygon(poly, LatLong)`, a ordem é preservada, mas a convenção de entrada do Maprix (UTM, CCW na UI) gera CCW no resultado.

**Proposta de correção (não implementar ainda):**
1. Em `shared/geo-core/src/shapefile.ts`, antes de chamar `buildShp`, rodar um helper `ensureClockwise(points)`:
   - Calcula shoelace
   - Se for CCW (sum < 0), reverte o array
   - Mantém o fechamento (primeiro = último)
2. Adicionar teste unitário com polígono conhecido em CCW e CW, verificando que ambos saem CW no output.

**Resolução aplicada:** helper puro `ensureClockwise(points)` exportado de `shared/geo-core/src/index.ts`, aplicado dentro de `gerarShapefileZip` antes de chamar `buildShp`. Lógica:
```ts
const sum = Σ (x2-x1)*(y2+y1)   // shoelace ESRI-style
return sum >= 0 ? points : [...points].reverse()
```
Idempotente (input CW retorna inalterado) e robusto (input já fechado ou aberto, ambos funcionam via índice circular).

**Validação:** mesmo polígono de teste (quadrado UTM 22S CCW):
- **Antes:** shoelace sum `−1.76e-6` → CCW
- **Depois:** shoelace sum `+1.76e-6` → CW ✅ (ordem invertida, primeiro=último preservado)

## ✅ GAP-02 — Campos `AREA_M2` e `PERIM_M` saem **zerados** no DBF — RESOLVIDO (commit pendente)

**Item do checklist:** 4.7, 4.8
**Evidência:** inspeção do record 1 mostra `"           0.000"` em ambos os campos. O DBF lê `poly.metadata?.area_m2 ?? 0` mas o Polygon vindo da UI **não** tem esses metadados — são calculados em runtime pelo `computeArea`/`computePerimeter`, mas nunca persistidos de volta ao objeto `Polygon`.

**Impacto:** o arquivo é entregue ao cartório com atributos zerados. Em qualquer validação manual o analista vê "Área=0" e rejeita ou, pior, registra como tal.

**Causa raiz:** separação entre:
- `backend/src/index.ts::/api/shapefile` — chama `gerarShapefileZip(polygon)` direto sem calcular metadata
- `shared/geo-core/src/shapefile.ts` — lê `poly.metadata?.area_m2` sem fallback de cálculo

**Proposta de correção:**
1. Dentro de `gerarShapefileZip`, após reprojetar para LatLong SIRGAS2000, **calcular** área e perímetro antes de escrever o DBF:
   ```ts
   const area = computeArea(projected);
   const perim = computePerimeter(projected);
   const projectedWithMeta = { ...projected, metadata: { ...projected.metadata, area_m2: area, perimetro_m: perim } };
   ```
2. Passar `projectedWithMeta` para `buildDbf`.
3. **Atenção**: área/perímetro devem refletir as **coordenadas originais** (em metros, sistema UTM), não as lat/long reprojetadas. `computeArea` já reprojeta internamente, então o número está certo. Mas conceitualmente o valor precisa ser explicado no memorial.

**Resolução aplicada:** `gerarShapefileZip` agora chama `computeArea(projected)` e `computePerimeter(projected)` imediatamente após a reprojeção e injeta os valores em `metadata` antes de passar para `buildDbf`.

**Validação:** quadrado 100×100m em UTM 22S SIRGAS2000 → no DBF:
- `AREA_M2 = 10033.067` (100×100 = 10000 m² + distorção de reprojeção esperada)
- `PERIM_M = 400.662` (4×100 = 400 m + curvatura geodésica)
- Valores idênticos aos que a UI exibe via `/api/convert` — consistência garantida.

## ✅ GAP-03 — Encoding do DBF indefinido; acentos truncados — RESOLVIDO (commit pendente)

**Itens do checklist:** 4.2, 7.1, 7.2, 7.3
**Evidência:**
- Byte 29 do header DBF = `0x00` (Language Driver indefinido)
- Função `asciiBytes()` faz `charCodeAt(i) & 0xff` — caracteres UTF-16 acima de 255 são truncados silenciosamente. Exemplo: "São Paulo" → `S` `?` `o` ` ` `P` `a` `u` `l` `o`, onde `?` é o byte 0xE3 (ã em Latin1) ou lixo dependendo da codepoint.

**Impacto:** qualquer string com ç, ã, é, ó, ú (ou seja, **todo nome brasileiro de município, proprietário, comarca**) é gravada incorretamente. Abre no QGIS como "Bras?lia", "Gr?o-Par?", etc. Em campo do cartório isso é imediatamente bloqueante — nomes próprios corrompidos em registro público são inaceitáveis.

**Proposta de correção (escolher uma):**

**Opção A (recomendada): ISO-8859-1 explícito.**
- Usar `Buffer.from(str, 'latin1')` (Node) ou um encoder manual que mapeia codepoints U+0000..U+00FF diretamente.
- Setar byte 29 do DBF para `0x03` (Windows ANSI / ISO-8859-1, o mais universalmente aceito por ferramentas GIS legadas).
- Criar arquivo `.cpg` com conteúdo `ISO-8859-1` ao lado dos outros.
- **Limitação:** caracteres fora do Latin1 (ex: emoji, chinês) viram `?`. Para nome brasileiro tradicional, cobre 100%.

**Opção B: UTF-8.**
- Usar `Buffer.from(str, 'utf8')`.
- Setar LDID para um valor que sinaliza UTF-8 (valores `0xF0`/`0x40` em alguns dialetos — **sem padrão oficial**).
- Criar `.cpg` com `UTF-8`.
- **Risco:** DBF clássico tem campo de tamanho em **bytes**, não caracteres. "Paraná" em UTF-8 são 7 bytes ("P a r a n à ©" = 7), em Latin1 são 6. Se o campo tem `len=10`, "Paraná" cabe em Latin1 mas em UTF-8 ainda cabe. Palavras mais longas podem cortar no meio de um caractere e gerar bytes inválidos.

**Recomendação:** **Opção A** para o MVP — padrão histórico SIGEF/INCRA é Latin1, e o risco de corte no meio de caractere é zero.

**Resolução aplicada:** Opção A (ISO-8859-1 / CP1252).
- `shared/geo-core/src/shapefile.ts`: função `asciiBytes` substituída por `latin1Bytes` com mapeamento direto `charCodeAt ≤ 0xFF → byte; caso contrário → '?'`.
- Byte 29 (LDID) do header DBF setado para `0x03` (Windows ANSI / CP1252).
- Arquivo `imovel.cpg` adicionado ao ZIP contendo literal `ISO-8859-1\n`.

**Validação:**
- Round-trip do helper com "São Paulo", "João Antônio Müller", "Açaí", "Coração" — ✅ todos idênticos.
- DBF record 1 com matrícula `"São-Paulo"`: bytes gravados `53 e3 6f 2d 50 61 75 6c 6f 20`, onde `0xE3` é o codepoint Latin-1 correto de `ã`.
- Arquivo `.cpg` presente no ZIP com 11 bytes (`ISO-8859-1\n`).
- LDID lido como `0x03` na re-inspeção.

## 🔴 GAP-04 — Nome de arquivo fixo `imovel.*` colide em envio múltiplo

**Item do checklist:** 1.3
**Evidência:** Manual ONR §3 diz que grupos diferentes de shapefile num mesmo lote precisam ter **nomes-base diferentes**. Nosso gerador sempre produz `imovel.shp/.shx/.dbf/.prj`.

**Impacto:** ao enviar dois polígonos no mesmo lote, o ONR rejeita por colisão. Não é catastrófico para um MVP (que gera um polígono por vez), mas é uma dívida que explode assim que o cliente tenta um batch.

**Proposta de correção:**
1. Aceitar um `baseName` opcional em `gerarShapefileZip(poly, options?: { baseName?: string })`:
   - Default: derivar de `poly.metadata?.matricula` com slug ASCII ("12.345" → `mat-12345`), ou fallback `imovel-{timestamp}`.
2. No backend, endpoint `/api/shapefile` aceita query ou body `?name=mat-12345`.
3. UI expõe campo "Nome-base do arquivo" com default automático.

## 🟠 GAP-05 — `.prj` sem `AUTHORITY["EPSG","4674"]`

**Item do checklist:** 5.4
**Evidência:** PRJ gerado é `GEOGCS["SIRGAS 2000",DATUM["SIRGAS_2000",SPHEROID["GRS 1980",6378137,298.257222101]],PRIMEM["Greenwich",0],UNIT["degree",0.0174532925199433]]`. Falta `AUTHORITY`.

**Impacto:** ferramentas como QGIS e ArcGIS **tentam inferir** o EPSG a partir do nome e do elipsoide, e geralmente acertam pra SIRGAS2000. Mas a identificação fica probabilística — o ideal é que o leitor reconheça deterministicamente "é EPSG:4674". Sem `AUTHORITY`, há risco de o sistema do ONR não bater com sua base interna e pedir reenvio.

**Proposta de correção:**
```
GEOGCS["SIRGAS 2000",
  DATUM["SIRGAS_2000",
    SPHEROID["GRS 1980",6378137,298.257222101,AUTHORITY["EPSG","7019"]],
    AUTHORITY["EPSG","6674"]],
  PRIMEM["Greenwich",0,AUTHORITY["EPSG","8901"]],
  UNIT["degree",0.0174532925199433,AUTHORITY["EPSG","9122"]],
  AXIS["Latitude",NORTH],
  AXIS["Longitude",EAST],
  AUTHORITY["EPSG","4674"]]
```

## 🟠 GAP-06 — Campo `MATRICULA` ausente (matrícula vai no `ID`)

**Item do checklist:** 4.6
**Evidência:** hoje a matrícula preenche o campo genérico `ID` (C,10). Se o ONR (ou os sistemas de análise da serventia) esperar um campo nominalmente chamado `MATRICULA`, fica parecendo um ID genérico sem semântica.

**Impacto:** incerto. O Manual v1.2 não exige. Mas sistemas de conciliação entre SIG-RI e SIGEF tendem a casar por nome de campo. Considerando que o **público-alvo é o cartório**, que lê o DBF no QGIS, um campo chamado `MATRICULA` é auto-explicativo; `ID` é ambíguo.

**Proposta de correção:** renomear/desdobrar.
- `MATRICULA` (C, 20) — número completo com hífens se houver
- `ID` pode ficar como auxiliar de ordenação (opcional)

## 🟠 GAP-07 — `.dbf` sem `PROPRIETAR`, `MUNICIPIO`, `UF`

**Itens do checklist:** 4.9, 4.10
**Evidência:** metadata do Polygon carrega esses campos (`matricula`, `proprietario`, `municipio`, `uf`) mas o DBF ignora tudo exceto `matricula`.

**Impacto:** mesmo raciocínio do GAP-06. Campos provavelmente **não obrigatórios** no upload (vão via formulário UI), mas **esperados** quando o DBF é aberto para inspeção humana ou importado por outros sistemas.

**Proposta de correção:** adicionar ao schema do DBF:
- `PROPRIETAR` (C, 50) — truncar em 50 chars
- `MUNICIPIO` (C, 40)
- `UF` (C, 2)
- `COMARCA` (C, 40) — opcional
- Requer alinhamento com Opção A do GAP-03 (Latin1) pra acentuação correta.

## 🟠 GAP-08 — Sem `.cpg` companion file

**Item do checklist:** 1.5
**Evidência:** ZIP só tem `.shp/.shx/.dbf/.prj`.

**Impacto:** sem `.cpg`, leitores precisam adivinhar encoding pelo LDID. Com o LDID também indefinido (GAP-03), alguns leitores caem em default do OS (ANSI no Windows BR ≈ Latin1, UTF-8 no Linux) e acertam por sorte.

**Proposta de correção:** escrever `.cpg` ao lado dos outros 4, com conteúdo exato da string do encoding escolhido no GAP-03. Um byte por linha sem BOM.

## 🟠 GAP-09 — Sem validação de geometria (self-intersection, vértices duplicados)

**Itens do checklist:** 6.3, 6.4, 6.5
**Evidência:** o gerador aceita qualquer array de pontos com length ≥ 3.

**Impacto:** polígono com auto-intersecção pode ser rejeitado pelo SIG-RI ou resultar em cálculo de área errado. Polígono com dois vértices iguais consecutivos (além do fechamento) vira degenerado.

**Proposta de correção:** adicionar em `gerarShapefileZip` uma validação prévia que use `@turf/boolean-valid` ou uma implementação caseira:
1. Polygon tem pelo menos 3 vértices únicos
2. Nenhum par de arestas adjacentes é colinear-degenerado
3. Nenhuma aresta cruza outra aresta (O(n²) é aceitável para imóvel único)
4. Área > threshold mínimo (ex: 1 m²)
5. Lançar erro descritivo com código (ex: `E_POLY_SELF_INTERSECT`) para a UI tratar.

## 🟢 GAP-10 — Header DBF sem eixos `AXIS` (WKT2)

**Item do checklist:** 5.5
**Impacto:** softwares WKT2-compliant (PROJ 6+, recentes QGIS/ArcGIS) aceitam WKT1 sem AXIS, mas podem questionar ordem de eixos. Hoje o Maprix produz lat/long no PRJ; em WKT1 a ordem default é lat-longitude (norte-east). QGIS trata bem, mas WKT2 explícito removeria qualquer dúvida.

**Proposta:** entra junto com GAP-05 (AUTHORITY). Mais robusto.

## O que NÃO está no plano dessa rodada

- **Validação semântica CNJ** (ex: saber se o ONR tem alguma validação de overlap/vizinhos que rejeita imóveis conflitantes). Isso é do domínio do sistema do ONR, não do gerador do cliente.
- **Campos específicos exigidos pela ITN 003/2025**. A ITN existe mas o texto integral não está publicamente acessível na data. Revisar quando estiver.
- **Exportação multi-polígono no mesmo ZIP**. Fora do escopo do MVP (1 imóvel por vez).
- **Validações para imóveis urbanos vs rurais** (possíveis schemas distintos). Hoje tratamos tudo igual. Se o ONR publicar schemas distintos, ramificar `gerarShapefileZip({ tipo: 'urbano' | 'rural' })`.
